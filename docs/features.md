## Ключевые User Flows

### 1. Посетитель знакомится с компанией

1. Открывает `/`.
2. Видит проекты, услуги, отзывы и последние статьи.
3. Переходит в `/projects/`, `/blog/`, `/documents/` или `/equipment/`.
4. Фильтрует каталоги по category/location.
5. Открывает детальную страницу нужной сущности.

### 2. Посетитель отправляет заявку

1. Заполняет форму на главной.
2. JS собирает данные и отправляет `POST /api/contact`.
3. Backend валидирует имя и наличие хотя бы одного контакта.
4. Заявка сохраняется в `contact_submissions`.
5. При настроенной почте отправляется email-уведомление.

### 3. Редактор работает через `/panel`

1. Логинится через `/panel/login`.
2. При включённой 2FA вводит TOTP-код.
3. Получает access token и refresh cookie.
4. Переходит в нужный модуль.
5. Работает со списком, формой, bulk-операциями и медиа.

### 4. Поддержка обрабатывает заявки

1. Открывает `/panel/contacts`.
2. Ищет запись по имени, email или телефону.
3. Открывает detail view заявки.
4. Отмечает заявку прочитанной или удаляет её.
5. При необходимости смотрит связанные действия в audit log.

## Роли И Доступ

Система уже содержит роли и permissions, но фактическая защита маршрутов сейчас в основном завязана на `is_admin`.

На практике есть два уровня:

- анонимный пользователь публичного сайта;
- администратор/сотрудник, проходящий через `/panel` или `/admin`.

Seed создаёт:

- `Администратор`;
- `Редактор`;
- `Наблюдатель`.

Но route-level enforcement по permissions пока не внедрён повсеместно.

## Ограничения Текущей Функциональности

| Ограничение | Комментарий |
|-------------|-------------|
| Публичный API минимален | Во внешний JSON вынесена только контактная форма |
| Полноценного route-level RBAC пока нет | Permissions есть в модели и UI, но не массово применяются в routes |
| Нет автотестов | Верификация в основном ручная |
| Две админки одновременно | `/panel` и `/admin` живут параллельно |
| Нет внешних интеграций CRM/analytics | Данные живут локально в БД и почте |
